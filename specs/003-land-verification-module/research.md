## Decision D-001 — `satellite_thumbnail_url` Field: SRS vs BSDD Discrepancy

**Unknown**: The BSDD (§5.3) `GET /api/v1/land/fetch-boundary` response does not
include a `satellite_thumbnail_url` field. The SRS (§6.5) response schema does
include it. Which is authoritative?

**Decision**: **SRS wins** per constitution Principle I and the governance
hierarchy (SRS > FDD > BSDD). The `satellite_thumbnail_url` field is part of
the fetch-boundary response and MUST be treated as a required field in the API
response type.

**Rationale**:  
The BSDD describes the backend implementation only. The SRS captures
agreed-upon product behaviour including data fields visible to the mobile
client. A field present in the SRS but absent from the BSDD is more likely a
BSDD documentation gap than a product decision. The user's explicit instruction
also confirms this field exists: _"GEE PNG URL comes from backend fetch-boundary
response field `satellite_thumbnail_url` — cache it in MMKV after first load."_

**Cache flow**:
1. `GET /api/v1/land/fetch-boundary` returns `satellite_thumbnail_url`.
2. Stored in `currentDraft.satellite_thumbnail_url` (Redux, not persisted to MMKV
   because `currentDraft` is blacklisted).
3. Displayed on `BoundaryConfirmScreen` as the map background `<Image>` source.
4. After successful `POST /api/v1/land/register`, a new `LandParcel` is
   constructed locally using data from the register response + `currentDraft`.
   `thumbnail_url` on the new parcel is set to `currentDraft.satellite_thumbnail_url`.
5. The new parcel is added to `land.parcels` via `addParcel()`.
6. `land.parcels` is MMKV-persisted — the URL is now cached offline automatically.
7. Subsequent online loads from `GET /api/v1/land/list` also return `thumbnail_url`
   which syncs the value.

**Note**: The `thumbnail_url` field in `GET /api/v1/land/list` response is the same
GEE PNG URL, just served from the backend's stored record after registration.

**Alternatives considered**:
- Trust only BSDD and omit the field — rejected: breaks BoundaryConfirmScreen
  satellite background and offline thumbnail caching.
- Fetch thumbnail URL from a dedicated endpoint after registration — rejected:
  unnecessary extra round-trip; the field is already available at fetch-boundary time.

## Decision D-002 — Gallery Picker Without `react-native-image-picker`

**Unknown**: `react-native-vision-camera` v4 provides camera capture only — it has
no built-in gallery/image library picker. The constitution explicitly forbids
`react-native-image-picker`. How does the "Upload from Gallery" action work?

**Decision**: Add `react-native-document-picker` as a new dependency. Use it to
launch an image-type file picker (`types.images` MIME filter) for gallery
selection. This package is NOT `react-native-image-picker` and satisfies the
constitution's "NOT react-native-image-picker" constraint.

**Rationale**:  
`react-native-document-picker` launches the Android native `Intent.ACTION_GET_CONTENT`
with MIME type `image/*`, allowing the farmer to select any image from the device
gallery or files app. It returns a result object containing `uri`, `type`, `name`,
and `size` — the `uri` can be used directly in a FormData upload, and `size` can
be validated for the 10 MB limit before upload.

**Install command** (pwsh, to be run during implementation):
```
npm install react-native-document-picker
```

**Gallery pick + FormData upload pattern**:
```typescript
import DocumentPicker, { types } from 'react-native-document-picker';

const result = await DocumentPicker.pickSingle({ type: [types.images] });
// result.uri, result.size, result.name available
```

**Alternatives considered**:
- `@react-native-camera-roll/camera-roll` — accesses the camera roll, but
  requires a separate custom picker UI (no native sheet). Rejected: too much
  bespoke UI work relative to document picker.
- `expo-image-picker` — requires Expo SDK. Rejected: project is RN CLI only.
- Custom native module (Android MediaStore) — rejected: overkill given
  `react-native-document-picker` already provides this through a proven library.

## Decision D-003 — BoundaryConfirmScreen: Image + MapView + Polygon Pattern

**Unknown**: How do we render a GEE Sentinel-2 PNG as a geographic background with a
GeoJSON polygon overlay correctly aligned to it, without using Google Maps satellite
tiles?

**Decision**: Stack an absolute-fill `<Image>` underneath a `<MapView mapType="none">`.
Set `MapView.region` to the bounding box computed from the GeoJSON coordinates
(coordinate extremes + 20% padding). Render `<Polygon>` inside the MapView from
the GeoJSON coordinates. The GEE PNG is generated by the backend for exactly the
same bounding box the MapView is initialised to, ensuring visual alignment.

**Implementation pattern**:
```tsx
// Compute bounds from GeoJSON
const coords = geojson.geometry.coordinates[0];
const lats = coords.map(c => c[1]);
const lngs = coords.map(c => c[0]);
const minLat = Math.min(...lats);
const maxLat = Math.max(...lats);
const minLng = Math.min(...lngs);
const maxLng = Math.max(...lngs);
const PADDING = 1.3; // 30% padding
const latDelta = (maxLat - minLat) * PADDING;
const lngDelta = (maxLng - minLng) * PADDING;

const region = {
  latitude: (minLat + maxLat) / 2,
  longitude: (minLng + maxLng) / 2,
  latitudeDelta: latDelta,
  longitudeDelta: lngDelta,
};

// JSX
<View style={{ flex: 1 }}>
  {/* Satellite PNG fills the container behind MapView */}
  <Image
    source={{ uri: satelliteThumbnailUrl }}
    style={StyleSheet.absoluteFill}
    resizeMode="cover"
  />
  {/* MapView with no tiles — transparent to show Image behind it */}
  <MapView
    style={StyleSheet.absoluteFill}
    mapType="none"
    region={region}
    scrollEnabled={false}
    zoomEnabled={false}
  >
    <Polygon
      coordinates={coords.map(c => ({ latitude: c[1], longitude: c[0] }))}
      fillColor="rgba(34,197,94,0.3)"
      strokeColor="rgba(34,197,94,0.9)"
      strokeWidth={2}
    />
  </MapView>
</View>
```

**Note**: The `<Image>` uses `StyleSheet.absoluteFill` which is the ONLY permitted
use of `StyleSheet` here — it's a geometry constant (not layout), not a
`StyleSheet.create` for screen layout. NativeWind cannot express `absoluteFill`
geometry as a utility class since the style object reference is required by RN
core. All other layout uses NativeWind classes.

**MapView background transparency**: On Android, `MapView` with `mapType="none"`
renders white by default. To make it transparent so the Image shows through,
set `MapView` style `backgroundColor: 'transparent'` and ensure the GEE Image
is positioned absolutely under the MapView in the Z-order.

**Alternatives considered**:
- `<Overlay>` from react-native-maps (custom image overlay with geographic bounds) —
  considered but requires the backend to return explicit `bounds` array with the PNG.
  The current SRS response does not include bounds separately, and computing them from
  GeoJSON extremes + setting MapView.region achieves the same alignment via the
  region coordination approach.
- Use a `<WebView>` with a Leaflet map — rejected: adds heavy dependency and WebView
  overhead for a single screen.

## Decision D-004 — Owner Name Mismatch Error Display

**Unknown**: The spec (User Story 2, §FR-009) shows the owner name mismatch error on
DocumentUploadScreen. But the error actually comes from `POST /api/v1/land/register`
which is called from BoundaryConfirmScreen. On which screen does the error appear?

**Decision**: The owner name mismatch error is displayed **on
BoundaryConfirmScreen**, not on DocumentUploadScreen. The error is received as an
HTTP 400 response from `POST /api/v1/land/register` when the farmer taps
"Yes, this is my land — Confirm". The inline error message is shown in the bottom
sheet below the confirm/reject buttons.

**Error message** (from SRS §6.5, exactly matching the spec FR-009):
> "The name on this document does not match your registered name. Please use the
> land document where you are listed as the owner."

**Rationale**:  
The FDD and SRS confirm farmer confirms OCR fields on DocumentUploadScreen, then
the app auto-calls fetch-boundary. The register call (which does name fuzzy matching)
happens only when the farmer taps "Confirm" on BoundaryConfirmScreen. Therefore
the 400 error from register MUST appear on BoundaryConfirmScreen, not on
DocumentUploadScreen. DocumentUploadScreen only triggers the OCR endpoint
(`verify-document`) which does not perform name matching (BSDD §5.3: "Do NOT
cross-check owner name here").

**Implementation**: BoundaryConfirmScreen maintains a `registerError: string | null`
in local state. On HTTP 400, set `registerError` to the `error` message from the
API response body. Render as a red `<Text>` inside the bottom sheet.

**Alternatives considered**:
- Navigate back to DocumentUploadScreen with error param — rejected: farmer is
  already past the boundary confirmation UI; navigating back discards the confirmed
  boundary state unnecessarily.

## Decision D-005 — GPS Best-Effort for `fetch-boundary` Query Params

**Unknown**: How does the app acquire GPS coordinates for `GET /api/v1/land/fetch-boundary`
query params `user_lat` and `user_lng`? What happens if GPS is unavailable?

**Decision**: Call `Geolocation.getCurrentPosition()` immediately when the farmer
taps "This is correct — Continue" on DocumentUploadScreen (after reviewing OCR
fields). Use a 5-second timeout. On success, pass `user_lat` and `user_lng` to the
fetch-boundary request. On timeout or error, proceed with the request **without**
`user_lat`/`user_lng` params — the backend handles missing GPS gracefully per SRS §6.5:
_"if GPS is unavailable, the call proceeds without coordinates."_

**Implementation**:
```typescript
const getGPS = (): Promise<{ lat: number; lng: number } | null> =>
  new Promise(resolve => {
    Geolocation.getCurrentPosition(
      pos => resolve({ lat: pos.coords.latitude, lng: pos.coords.longitude }),
      () => resolve(null),
      { timeout: 5000, enableHighAccuracy: false },
    );
  });
```

**Note**: `enableHighAccuracy: false` is suitable here — this GPS is for WMS bbox
construction, not precise farmer location. A rough centroid within a few hundred
metres is sufficient for the ±0.01 degree bounding box used by the WMS.

**Alternatives considered**:
- Block the confirm action until GPS is acquired — rejected: rural areas can have
  poor GPS signal; blocking would cause the flow to stall indefinitely.
- Use the last-known position — rejected: last-known GPS could be from a different
  location entirely; best-effort fresh read is safer.

## Decision D-006 — Image File Size Validation (≤ 10 MB)

**Unknown**: The spec requires client-side validation that the captured/selected
image is ≤ 10 MB before uploading. `react-native-fs` is not installed. How do we
get the file size?

**Decision**:
- **Camera capture (VisionCamera)**: Configure `Camera.takePhoto()` with
  `qualityPrioritization: 'speed'` and target output size. VisionCamera v4 `.takePhoto()`
  returns a `PhotoFile` with a `path` (file URI). Use `react-native`'s bundled
  `fetch` or — more reliably — read the file size via the `XMLHttpRequest` blob
  technique. Simpler: use VisionCamera's `enableVideoStabilization` + compression
  quality so output is typically well under 5 MB for a document. Additionally validate
  via `fetch(path).then(r => r.blob()).then(b => b.size)`.
- **Gallery picker**: `react-native-document-picker` result includes `size` in bytes.
  Check `result.size > 10 * 1024 * 1024` before upload.

**Error message** (verbatim from spec):
> "Image is too large. Please take a clearer, smaller photo."

**Simpler alternative for camera**: `VisionCamera.takePhoto({ quality: 85 })` with
`photoQualityBalance: 'speed'` produces JPEG files typically 1–4 MB for a document
at normal resolution. Client-side size check via `Blob.size` is a safeguard for
edge cases. For camera output, size guard is secondary; gallery-selected images
are the higher risk (user may select a high-res screenshot).

**Final pattern**:
```typescript
// For gallery (document picker result)
if (result.size && result.size > 10 * 1024 * 1024) {
  setError('Image is too large. Please take a clearer, smaller photo.');
  return;
}
// For camera (fetch blob)
const response = await fetch(`file://${photo.path}`);
const blob = await response.blob();
if (blob.size > 10 * 1024 * 1024) {
  setError('Image is too large. Please take a clearer, smaller photo.');
  return;
}
```

## Decision D-007 — Offline Detection Without `@react-native-community/netinfo`

**Unknown**: `@react-native-community/netinfo` is absent from `package.json`. How
does the app show an offline banner and preserve draft data when a network call fails
due to no connectivity?

**Decision**: Use axios error inspection in a single catch block. If axios throws
an error with no response (`!error.response`), treat it as offline. Store a local
`isOffline: boolean` state in each screen. Show a banner when `isOffline` is true.
Draft data is preserved in Redux `currentDraft` automatically (cleared only on
explicit user action or successful registration, not on error).

**Additionally**: Add `@react-native-community/netinfo` as a new dependency at
implementation time. It provides a `useNetInfo()` hook that gives live connectivity
state, enabling the banner to be shown proactively (before a call fails) and dismissed
automatically when connectivity restores.

**Recommended install** (during implementation):
```
npm install @react-native-community/netinfo
```

**Pattern (with netinfo)**:
```typescript
import NetInfo from '@react-native-community/netinfo';
const netInfo = await NetInfo.fetch();
if (!netInfo.isConnected) {
  setOffline(true);
  return; // Don't attempt the API call
}
```

**Alternatives considered**:
- Error-only detection (no netinfo) — functional but reactive only; farmer sees
  a loading spinner before the error banner appears, which is a worse UX.
- Re-implementing background connectivity polling — rejected: exactly what
  netinfo already does, no point reinventing.

## Decision D-008 — `LandParcel.boundary_geojson` Nullability Gap

**Unknown**: The existing `landSlice.ts` declares `boundary_geojson: GeoJSONPolygon`
as non-nullable. But `GET /api/v1/land/list` does not return GeoJSON in its response.
Parcels loaded from the list API will have no GeoJSON data. Is this a type gap?

**Decision**: **Yes, this is a gap.** Change `boundary_geojson: GeoJSONPolygon` to
`boundary_geojson: GeoJSONPolygon | null` in the `LandParcel` type in `landSlice.ts`.

**Rationale**:  
When `LandListScreen` loads and calls `GET /api/v1/land/list`, the API response
parcels do not include GeoJSON. Constructing a `LandParcel` from the API response
would require assigning `null` to `boundary_geojson`. The field is only populated
during the registration flow when the device has the geometry from `fetch-boundary`.

GeoJSON is available for parcels registered in the same session (via `currentDraft.boundary`
used to construct the parcel after `POST /api/v1/land/register`). For subsequent
sessions, GeoJSON can be retrieved if needed for audit start (future feature scope).

**Required change** (already scaffolded, minor fix):
```typescript
// landSlice.ts — change from:
boundary_geojson: GeoJSONPolygon;
// to:
boundary_geojson: GeoJSONPolygon | null;
```

## Decision D-009 — Document Image Upload (`multipart/form-data`)

**Unknown**: BSDD §5.3 specifies `POST /api/v1/land/verify-document` accepts
`multipart/form-data` with an `image` file field. How does the RN app send a file
captured by VisionCamera or picked from the device gallery as multipart?

**Decision**: Use RN's `FormData` with a file descriptor object understood by the
RN HTTP layer (Fresco/OkHttp on Android). Pass `formData` to `apiClient.post()`
with `Content-Type: 'multipart/form-data'`.

**Pattern**:
```typescript
const uploadDocument = async (uri: string, mimeType = 'image/jpeg') => {
  const formData = new FormData();
  formData.append('image', {
    uri,
    type: mimeType,
    name: 'document.jpg',
  } as unknown as Blob);

  return apiClient.post('/land/verify-document', formData, {
    headers: { 'Content-Type': 'multipart/form-data' },
    timeout: 60_000, // OCR can be slow on server side
  });
};
```

**URI normalisation**:
- VisionCamera `photo.path` needs `file://` prefix: `file://${photo.path}` on iOS;
  on Android it typically does not need the prefix but adding it is harmless.
- `react-native-document-picker` returns Android content URIs (`content://`) which
  are directly usable in FormData on Android without conversion.

**Alternatives considered**:
- Base64 encode and send JSON body — some endpoints accept this, but BSDD explicitly
  specifies `multipart/form-data`. Following the contract is non-negotiable.
- Use `axios` streaming upload — not supported in RN's bundled `XMLHttpRequest`
  polyfill; FormData is the correct mechanism.
