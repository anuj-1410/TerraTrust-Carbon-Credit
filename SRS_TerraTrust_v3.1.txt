SECTION 1: INTRODUCTION
================================================================================

1.1 PURPOSE
-----------
This document defines every functional and technical requirement for
TerraTrust-AR — a decentralised carbon credit verification system for
Indian farmers. It supersedes all previous SRS versions.

This document is written so that any developer can pick it up and build
the complete system without asking further architecture questions.
All design decisions made across all discussion sessions are fully
incorporated here. Where the old SRS contradicts this document, this
document takes precedence.

1.2 WHAT THE SYSTEM DOES (Plain Language)
------------------------------------------
Farmers grow trees on their land. Trees absorb carbon dioxide from the
atmosphere. This absorbed carbon has measurable monetary value in global
carbon markets. However, to earn carbon credits, the amount of carbon
stored must be independently and tamper-proofably verified.

TerraTrust-AR automates this entire verification process:

  Step 1 — Proves the farmer legally owns the land (government records).
  Step 2 — Measures representative trees using smartphone AR — without
            requiring any technical knowledge from the farmer.
  Step 3 — Uses 6 layers of satellite data to estimate biomass across
            the entire farm, calibrated by the AR measurements.
  Step 4 — Issues tamper-proof Carbon Ton Tokens (CTT) on the Polygon
            blockchain, with permanent evidence stored on IPFS.

1.3 SCOPE
---------
Input  : Government land document (7/12 Extract or Record of Rights),
         AR tree scans from farmer's Android smartphone
Process: PaddleOCR document reading, LGD API + BhuNaksha WMS boundary
         fetching, ARCore trunk diameter measurement, 6-layer satellite
         fusion, XGBoost biomass modelling, Polygon blockchain minting
Output : Verified Carbon Ton Tokens (CTT) on Polygon blockchain +
         Permanent IPFS audit certificate

Out of scope for v1: Carbon credit trading platform, market pricing
engine, buyer-facing web dashboard.

1.4 INTENDED USERS
------------------
Primary   : Indian agroforestry farmers growing trees on agricultural land.
            May have limited smartphone literacy. Must complete the process
            independently with no technical assistance.
Secondary : Carbon credit buyers (companies offsetting emissions).
            They verify credits by reading IPFS certificates and
            PolygonScan blockchain records.
Admin     : TerraTrust platform operator (your team). Monitors audits,
            manages Oracle wallet, handles escalations.

1.5 KEY DESIGN PRINCIPLES
--------------------------
* The farmer must never understand what is happening technically.
  Every complex operation happens on the backend silently.
* Every screen instruction uses simple, regional-language-friendly
  wording. No technical terms shown to farmer.
* The system must work on any Android phone — including budget devices
  below Rs 8,000 released after 2018.
* All infrastructure used must be free at MVP scale.
* Every verification step must produce tamper-proof, independently
  auditable evidence.
* Offline-first for all field operations. Scanning, GPS navigation,
  and data saving must work without internet. Upload happens when
  connectivity is restored.
* The official government land boundary — not a farmer-drawn polygon —
  is the only accepted boundary. Carbon markets require this.

2.2 DATA FLOW SUMMARY
----------------------
1. Farmer uploads 7/12 document photo.
   → PaddleOCR extracts Survey Number, Owner Name, Village, Taluka,
     District from the document image.
   → Backend calls Maharashtra LGD API to get official location codes.
   → Backend calls BhuNaksha WMS GetFeatureInfo to get official GeoJSON
     polygon of the survey number.
   → Polygon stored in Supabase PostGIS as the permanent land boundary.
   → Every future operation is restricted to pixels inside this boundary.

2. Backend runs Sentinel-2 NDVI analysis on the verified land boundary.
   → Identifies distinct vegetation density zones (high / medium / low).
   → Places one sampling point inside each zone.
   → Calculates a linear walking path through all zone points.
   → Returns GPS coordinates of zone centres to the app.

3. Farmer walks to each zone and scans trees.
   → ARCore measures Diameter at Breast Height (DBH) in cm.
   → TensorFlow Lite identifies species on-device (no internet needed).
   → GEDI satellite provides tree height from space (primary source).
   → AR anchor height measurement used as fallback if no GEDI footprint.
   → Each scan: species + DBH + GPS + evidence photo saved locally.

4. Farmer submits all scans.
   → Backend runs 6-layer satellite fusion (Celery background task):
       NISAR L-band (primary from June 2026) + Sentinel-1 C-band +
       Sentinel-2 optical + GEDI canopy height + SRTM topography.
   → For each scanned tree: Chave's allometric equation calculates AGB.
   → XGBoost trained on AR ground truth predicts biomass for every
     10m × 10m pixel in the land boundary.
   → Total biomass = sum of all pixel predictions.

5. Delta from last year calculated → CO2 credits determined.
   → Evidence packaged as JSON and uploaded to IPFS via NFT.Storage.
   → Smart contract mints CTT tokens to farmer's blockchain wallet.
   → IPFS CID and transaction hash stored permanently in database.

2.3 SYSTEM ARCHITECTURE PATTERN
---------------------------------
The backend follows a Modular Monolithic Architecture — all services
(Auth, Land, Fusion, Blockchain) live in one Python deployment but are
structured as separate modules. Each module can be extracted to its own
microservice later without rewriting.

Asynchronous event-driven for heavy operations: the API immediately
returns a response. Satellite computation and blockchain minting run
in the background via Celery workers. The mobile app polls for results.

SATELLITE DATA (All free via Google Earth Engine unless noted)
--------------------------------------------------------------
NISAR L-band  : Primary SAR from June 2026. Joint NASA-ISRO mission.
                Launched July 30, 2025. L-band (~24cm wavelength)
                penetrates full tree canopy. 5-10m resolution.
                Access: asf_search Python library (Alaska Sat. Facility)
                Product type: GCOV (Geocoded Covariance)
                Data available: October 2025. Fully calibrated: June 2026.
                Phase 1 (until June 2026): Development/testing only.
                Phase 2 (June 2026+): Primary production SAR.

Sentinel-1 C  : Supplementary SAR (primary until June 2026).
                ESA satellite, operational since 2014. C-band (~6cm).
                Works through clouds — critical for Indian monsoon.
                10m resolution. Updated every 6 days. Free on GEE.
                Bands used: VH, VV, VH/VV ratio.

Sentinel-2    : Optical multispectral. ESA. 10m resolution. 5-day cycle.
                Indices used: NDVI (B8, B4), EVI (B8, B4, B2),
                Red-Edge (B7). Cannot penetrate clouds — use annual median.
                Also used for: zone NDVI analysis, Sentinel-2 true colour
                PNG for BoundaryConfirmScreen and land thumbnails.

GEDI          : NASA LiDAR from International Space Station.
                Provides actual canopy height (rh98 metric).
                25m footprint diameter. Funded until January 2031.
                Primary source of tree height. AR height = fallback.
                Available on GEE: LARSE/GEDI/GEDI02_A_002_MONTHLY

SRTM          : NASA Shuttle Radar Topography Mission. Global 30m DEM.
                Used for: terrain effect correction on radar signal,
                slope/aspect inputs for biomass modelling.
                Available on GEE: USGS/SRTMGL1_003

Establish who the farmer is before any land or tree operations.

5.2 SCREENS AND FLOWS
-----------------------

SCREEN: SplashScreen

                     If session valid and KYC complete → navigate to HomeScreen

SCREEN: LoginScreen

                     Invalid: show "Please enter a valid 10-digit number"

                     Supabase sends 6-digit OTP via SMS automatically.
  On success:        Navigate to OTPScreen.

SCREEN: OTPScreen
  What farmer sees:  Six individual digit input boxes.
                     Instruction text: "Enter the 6-digit code sent to
                     [farmer's phone number]"
                     Timer: "Resend in 28s" counting down.
                     After timer reaches 0: "Resend OTP" link appears.
  Behaviour:         Each box auto-focuses the next box when a digit is entered.
                     Backspace moves focus back to previous box.
  On verify:         Call supabase.auth.verifyOtp({ phone, token, type: 'sms' })
  On success:        Check user profile in Supabase users table.
                     If no wallet_address on record → run silent wallet

SCREEN: KYCScreen

                       Full Name — "Enter your name exactly as written on

                       Aadhaar Number — 12-digit field
                     One button: "Continue"
  Why name matters:  This name is matched against the OCR-extracted owner
                     name from the 7/12 document in Module 2. If they do
                     not match, land registration is rejected.
  On submit:         POST /api/v1/auth/kyc to backend.
                     Backend stores: full_name, SHA-256 hash of Aadhaar
                     (never plain text), sets kyc_completed = true.
                     Navigate to HomeScreen.

5.3 SILENT WALLET CREATION
----------------------------
This process happens automatically after the first OTP verification when
no wallet address exists for this farmer. The farmer sees nothing and does
not interact with it.

Step 1: ethers.js generates a cryptographically random private key and
        derives the public wallet address from it.

Step 2: The private key is stored in react-native-keychain, which uses
        Android Keystore hardware-backed encrypted storage.
        The key NEVER leaves the device.
        It is NEVER stored in Redux state.
        It is NEVER stored in AsyncStorage or MMKV.
        It is NEVER sent to any server.
        It is NEVER logged in any console or error report.

Step 3: Only the public wallet address is sent to the backend and stored
        in the Supabase users table (wallet_address column).

Step 4: When the farmer earns Carbon Ton Tokens, the backend Oracle mints
        them to this address. The farmer sees their balance as a plain
        number. They never see a wallet address or any blockchain UI.

Code reference (services/wallet.ts):
  import { ethers } from 'ethers';
  import * as Keychain from 'react-native-keychain';

  export const createFarmerWallet = async (): Promise<string> => {
    const wallet = ethers.Wallet.createRandom();
    await Keychain.setGenericPassword(
      'wallet_private_key',
      wallet.privateKey,
      {
        accessControl:
          Keychain.ACCESS_CONTROL.BIOMETRY_ANY_OR_DEVICE_PASSCODE,
        service: 'terratrust_wallet'
      }
    );
    return wallet.address; // Only the public address is returned and used
  };

5.4 BACKEND: POST /api/v1/auth/kyc
-------------------------------------

  {
    "full_name": "Ramesh Shankar Patil",
    "aadhaar_number": "XXXXXXXXXXXX"   // 12 digits

Response:
  { "status": "success" }

SECTION 6: MODULE 2 — LAND VERIFICATION (THE DIGITAL FENCE)
================================================================================

6.1 PURPOSE
-----------
Prove that the farmer legally owns the specific land they are claiming.
Extract the exact government-surveyed boundary coordinates.
Store this verified boundary — every future satellite calculation and
AR scan is restricted to pixels and GPS points inside this boundary.

The boundary MUST come from government records. Any farmer-drawn polygon
is not acceptable for carbon credit verification under any carbon market
standard (Verra, Gold Standard, etc.).

IMPORTANT: The previous design (v2.x) included "Draw Mode" (farmer taps
polygon corners on map) and "Walk Mode" (GPS-tracked walking). Both of
these are REMOVED. They are NOT implemented as farmer-facing features.
An administrative polygon drawing tool may exist as a hidden debug
feature only.

6.2 THE 3-LAYER STATE ROUTER SYSTEM
-------------------------------------
Different Indian states have different levels of digital infrastructure.
The backend automatically detects the farmer's state and routes to the
appropriate boundary fetching method. The farmer sees the same screens
regardless of which layer is used internally.

LAYER 1 — Maharashtra LGD API + BhuNaksha WMS (Fully Automatic)
  Coverage: All 36 districts of Maharashtra. Currently the only state
            with confirmed working integration.
  How it works:
  1. OCR extracts Survey Number, Owner Name, Village, Taluka, District.
  2. Backend calls Maharashtra LGD REST API at http://115.124.105.220/API/
     to get official numeric location codes for the district/taluka/village.
  3. Backend constructs the gis_code using those codes.
  4. Backend calls BhuNaksha WMS GetFeatureInfo to get the official
     GeoJSON polygon for that survey number.
  5. Result: verified polygon returned to app.

LAYER 2 — BhuNaksha Web Scraping (Fallback if Layer 1 WMS fails)
  Used when: WMS endpoint times out, returns error, or empty response.
  How it works:
  Selenium/Playwright headless browser navigates BhuNaksha portal.
  Selects District, Taluka, Village from dropdowns.
  Clicks the survey number on the rendered map.
  Extracts GeoJSON polygon from the map canvas.
  Returns same GeoJSON format as Layer 1.
  States supported: Maharashtra + other states using NIC BhuNaksha
  software (Karnataka, Rajasthan, UP, MP, AP, Telangana, Tamil Nadu,
  Gujarat, Punjab, Haryana, Bihar, Odisha, West Bengal, Kerala, Assam,
  Himachal Pradesh, Uttarakhand).

LAYER 3 — Manual Guided Upload (Universal Fallback)
  Used when: Layer 1 and Layer 2 both fail, OR state not yet integrated.
  How it works:
  App shows the farmer step-by-step instructions to download their land
  map from the government portal and upload it as an image.
  Backend uses OpenCV contour detection to extract the polygon boundary
  from the uploaded map image (using the dark boundary lines on the map).
  Then georeferences the pixel coordinates to lat/lng using the scale bar
  and coordinates visible on the BhuNaksha printed map.
  Returns same GeoJSON format as Layers 1 and 2.

State Router Config (backend/services/land_boundary_service.py):
  STATE_WMS_CONFIG = {
    "Maharashtra": {
      "lgd_api": "http://115.124.105.220/API",
      "wms_url": "https://mahabhunakasha.mahabhumi.gov.in/WMS",
      "layer": 1
    },
    "Karnataka": {
      "lgd_api": "http://115.124.105.220/API",
      "wms_url": "https://landrecords.karnataka.gov.in/GeoServer/WMS",
      "layer": 2
    }
    # Extend as states are tested and confirmed working
  }

6.3 STEP-BY-STEP LAND VERIFICATION FLOW
-----------------------------------------

STEP 1 — Farmer Uploads Land Document
  Screen: DocumentUploadScreen
  What farmer sees:
    Two buttons: "Take Photo" and "Upload from Gallery"
    Instruction: "Take a clear photo of your 7/12 Extract or
    Record of Rights document. Make sure all text is visible."
  Camera library: react-native-vision-camera v4
  After capture:
    Preview shown with overlay checking photo quality.
    "Confirm and Process" button.
  Processing state: Spinner with text: "Reading your document..."

STEP 2 — OCR Processing (Backend)
  OCR Engine: PaddleOCR PP-OCRv5 with devanagari_PP-OCRv5_mobile_rec
  NOTE: The old SRS specified Tesseract. This is WRONG in v3.1.
        PaddleOCR is the correct engine. Do not use Tesseract.

  Preprocessing (OpenCV — simplified because PaddleOCR handles orientation):
    a) Convert to greyscale
    b) Apply adaptive thresholding (handles uneven lighting)
    c) Denoise (fastNlMeansDenoising)
    NOTE: Deskew is NOT done manually. PaddleOCR's
    use_doc_orientation_classify and use_doc_unwarping flags handle
    rotation and perspective distortion internally.

  PaddleOCR initialisation (module-level, not per-request):
    from paddleocr import PaddleOCR
    ocr_engine = PaddleOCR(
        text_recognition_model_name="devanagari_PP-OCRv5_mobile_rec",
        use_doc_orientation_classify=True,
        use_doc_unwarping=True,
        use_textline_orientation=True,
        device="cpu"
    )

  Field extraction regex patterns (handle English + Marathi labels):
    Survey Number:
      r'(?:Survey|Gat|Gut|सर्व्हे|गट)\s*[Nn]o\.?\s*[:\-]?\s*(\d+(?:/\d+)?)'
    Owner Name:
      r'(?:Name of Owner|Owner|मालकाचे नाव|धारकाचे नाव)\s*[:\-]?\s*'
      r'([A-Z][a-zA-Z\s]+|[\u0900-\u097F\s]+)'
    Village:
      r'(?:Village|गाव)\s*[:\-]?\s*([A-Z][a-zA-Z\s]+|[\u0900-\u097F\s]+)'
    Taluka:
      r'(?:Taluka|तालुका)\s*[:\-]?\s*([A-Z][a-zA-Z\s]+|[\u0900-\u097F\s]+)'
    District:
      r'(?:District|जिल्हा)\s*[:\-]?\s*([A-Z][a-zA-Z\s]+|[\u0900-\u097F\s]+)'

  On success: App shows extracted fields to farmer for confirmation.
    Survey Number: [extracted]
    Owner Name: [extracted]
    Village: [extracted]
    Taluka: [extracted]
    District: [extracted]
    Two buttons: "This is correct — Continue" and "Try Again"

  Name match validation:
    Backend (at land registration step, not here):
    Compare extracted owner_name against user's KYC full_name using
    fuzzy string matching (80%+ similarity using difflib or rapidfuzz).
    This allows for OCR errors and middle name differences.
    If below 80%: Return 400 with message "The name on this document
    does not match your registered name. Please use the land document
    where you are listed as owner."

STEP 3 — Determine Layer and Fetch Boundary (Backend)
  Backend checks STATE_WMS_CONFIG for farmer's state.
  Tries Layer 1. On failure tries Layer 2. On failure returns
  { "status": "manual_required" } and app shows Layer 3 guide.

STEP 4 — LGD Code Lookup (Layer 1 only)
  Maharashtra LGD API at http://115.124.105.220/API/

  API Call 1: Get all districts:
    POST http://115.124.105.220/API/GetAllDistricts
    Body: {}
    Response: [{"distcode": 523, "distname_eng": "Pune", ...}]
    Action: Match extracted district name to get distcode.

  API Call 2: Get talukas of that district:
    POST http://115.124.105.220/API/GetTalukasOfDistrict
    Body: {"distcode": 523}
    Response: [{"talukacode": 4521, "talukaname_eng": "Haveli", ...}]
    Action: Match extracted taluka name to get talukacode.

  API Call 3: Get villages:
    POST http://115.124.105.220/API/GetVillagesOfDistrictAndTaluka
    Body: {"distcode": 523, "talukacode": 4521}
    Response: [{"villagecode": 20047, "villagename_eng": "Kharadi", ...}]
    Action: Match extracted village name to get villagecode.

  Cache all LGD codes in Redis (expire: 30 days).
  LGD codes change rarely. Caching is safe and reduces API load.

STEP 5 — Construct gis_code
  The gis_code is the NIC internal identifier for each plot in BhuNaksha.
  Pattern:
    "RVM" + region_code + division_code + "27" (Maharashtra state code)
    + taluka_code_padded_to_4_digits
    + village_code_padded_to_6_digits
    + survey_number_padded_to_6_digits

  def construct_gis_code(distcode, talukacode, villagecode, survey_number):
      state_code = "27"    # Maharashtra
      survey_padded = str(survey_number).zfill(6)
      village_padded = str(villagecode).zfill(6)
      taluka_padded = str(talukacode).zfill(4)
      return f"RVM05{state_code}{taluka_padded}{village_padded}{survey_padded}"

  NOTE: This exact pattern must be validated against 5-10 known survey
  numbers from Pune district during Week 3 of development. The pattern
  may vary by division code — test before relying on it.

STEP 6 — WMS GetFeatureInfo (Get Official Polygon Coordinates)
  Build bounding box: ±0.01 degrees around farmer's current GPS location.
  This creates approximately a 1km × 1km box around the farmer.
  Convert farmer GPS to pixel coordinates within the bounding box.

  WMS call:
    GET https://mahabhunakasha.mahabhumi.gov.in/WMS?
        SERVICE=WMS
        &VERSION=1.3.0
        &REQUEST=GetFeatureInfo
        &QUERY_LAYERS=PLOT_BOUNDARY
        &INFO_FORMAT=application/json
        &gis_code=[constructed_code]
        &BBOX=[lng-0.01],[lat-0.01],[lng+0.01],[lat+0.01]
        &WIDTH=800
        &HEIGHT=600
        &CRS=EPSG:4326
        &I=[pixel_x]
        &J=[pixel_y]

  On success: Response contains GeoJSON with polygon coordinates.
  On failure (timeout, empty, error): Fallback to Layer 2 scraping.

STEP 7 — Farmer Confirmation (BoundaryConfirmScreen)
  What farmer sees:
    Full-screen satellite photograph of their land with the official
    boundary polygon drawn on top as a semi-transparent green overlay.

  HOW THE SATELLITE IMAGE IS GENERATED:
    This screen does NOT use Google Maps satellite tiles (costs money).
    The backend generates a static Sentinel-2 true colour PNG image of
    the farm area using Google Earth Engine:
      region = ee.Geometry.Polygon(geojson_coords)
      s2 = ee.ImageCollection('COPERNICUS/S2_SR_HARMONIZED')
               .filterBounds(region)
               .filter(ee.Filter.lt('CLOUDY_PIXEL_PERCENTAGE', 20))
               .median()
      rgb = s2.select(['B4', 'B3', 'B2'])
      # Export as PNG with getThumbURL()
    The backend returns the PNG URL to the app.
    App loads PNG as background image using react-native-maps with
    mapType="none", then draws the GeoJSON polygon overlay with <Polygon>.
    Why: Sentinel-2 imagery is free with no per-request cost, often more
    recent than Google Maps satellite for rural India, and clearly shows
    agricultural field boundaries.

  Bottom information card shows:
    Survey Number: [value]
    Area: [calculated from polygon, in acres]
    Owner Name: [OCR-extracted value]
  Two buttons:
    "Yes, this is my land — Confirm" (green, primary)
    "This boundary is wrong — Report and Try Again" (red, small text)

  On confirm: POST /api/v1/land/register (see API section).
  On reject:  Navigate back to DocumentUploadScreen.

STEP 8 — Store Verified Boundary (Backend)
  Validation before saving:
    ST_IsValid(ST_GeomFromGeoJSON('[geojson]')) — must return true.
    Area must be between 0.1 hectares and 100 hectares.
    User must not already have this survey number registered.
    Owner name fuzzy match must pass (≥80% similarity).

  Insert into land_parcels table. Area auto-calculated by PostGIS trigger.
  verification_method stored: 'WMS_AUTO', 'SCRAPE', or 'MANUAL'.
  (This matters for audit credibility scoring — WMS_AUTO is highest.)

================================================================================
SECTION 7: MODULE 3 — AR TREE SCANNING (GROUND TRUTH COLLECTION)
================================================================================

7.1 PURPOSE
-----------
Collect precise physical measurements of representative trees.
These measurements are the ground truth that calibrates the satellite
biomass model for this specific farm.

IMPORTANT: AR scanning MEASURES DBH (diameter at breast height) and
Species only. It does NOT measure height as primary source.

HEIGHT SOURCE PRIORITY:
  Primary: NASA GEDI satellite rh98 canopy height (from space).
  Fallback: AR anchor height (farmer taps base, tilts to top) — used
            only when GEDI has no laser footprint at that GPS location.
  This was changed from v2.x where AR measured height as the primary
  method. GEDI is more consistent and eliminates farmer measurement error.

7.2 SATELLITE-GUIDED ZONE SAMPLING
------------------------------------
Before the farmer walks to the field, the backend pre-analyzes the farm
using Sentinel-2 NDVI to identify distinct vegetation zones.

Process (runs when farmer taps "Start Audit"):
  1. Backend fetches farm boundary from land_parcels table.
  2. GEE fetches Sentinel-2 NDVI for the boundary.
  3. NDVI percentile analysis identifies 3 vegetation density zones:
       Low density:    NDVI < 25th percentile
       Medium density: NDVI between 25th and 75th percentile
       High density:   NDVI > 75th percentile
  4. Backend finds the centroid of each zone cluster.
  5. Calculates plot radius based on land area:
       Area ≤ 0.4 ha (≤1 acre):   radius = 7 metres per zone
       Area ≤ 1.2 ha (≤3 acres):  radius = 9 metres per zone
       Area > 1.2 ha (>3 acres):  radius = 11 metres per zone
  6. Calculates a LINEAR walking path through all zone centres
     (nearest neighbour order — like a transect, not random).
  7. Checks GEDI footprint availability within each zone.
     Returns gedi_available flag per zone to the app.
  8. Saves zones to sampling_zones table.
  9. Returns zone GPS coordinates + walking path to app.

Zone count by farm size:
  Up to 1 acre:    2-3 zones
  1 to 3 acres:    3-4 zones
  3 to 10 acres:   4-6 zones
  Above 10 acres:  6-8 zones (split into 2 sessions on separate days)

Farmer scans every tree inside the zone radius circle.
Minimum trees required: 3 trees per zone. Total minimum: 9 trees.
Maximum trees: 5 trees per zone (stop and move to next zone after 5).
If fewer than 3 eligible trees in a zone: zone marked low-density and

7.3 THE 3-TIER AR MEASUREMENT SYSTEM
--------------------------------------
The app runs ArCoreApk.checkAvailability() on startup and stores the
result in Redux state (audit.arTier: 1, 2, or 3). This determines the
measurement method for the entire audit session.

TIER 1 — Full Depth AR (Hardware depth sensor phones)
  Applicable to: Phones with Time-of-Flight (ToF) hardware sensor, OR
                 phones with powerful enough processors for dense software
                 depth estimation via ARCore Depth API.
                 ARCore Depth API returns SUPPORTED_INSTALLED.
  What farmer does: Hold phone 1-2 metres from tree trunk, facing trunk
                    directly. Tap "Measure". Hold still for 3 seconds.
  What happens:    ARCore captures 16-bit raw depth map.
                   Kotlin identifies trunk point cloud in centre of frame.
                   RANSAC algorithm fits a 3D cylinder to those points.
                   Diameter = best cylinder radius × 2.
  Validation:      If RANSAC fit error > 5%: "Move closer to the tree
                   and hold still, then try again."
                   If point count < 50: "Not enough depth data. Move
                   closer and try again."
                   If confidence < 0.7: return low_confidence flag.
  Accuracy:        ±2-3 cm error. Green "High Precision" badge shown.

TIER 2 — Motion-Based AR (ARCore phones without hardware depth)
  Applicable to: Phones where ARCore is available but hardware depth
                 sensor is not. ARCore returns SUPPORTED_INSTALLED but
                 depth mode returns SLAM_ONLY.
  What farmer does: Phone shows animated arrow. Farmer moves phone slowly
                    left and right in a small arc for 5 seconds while
                    keeping the trunk centred on screen.
  What happens:    ARCore SLAM builds a sparse 3D point cloud from motion
                   parallax. RANSAC cylinder fitting runs on this cloud.
  Accuracy:        ±4-5 cm error. Blue "Standard Precision" badge shown.

TIER 3 — Manual String Measurement (Any Phone, Universal Fallback)
  Applicable to: Phones without ARCore support. Also offered to elderly
                 farmers who are uncomfortable with AR. Also shown
                 automatically after 3 consecutive Tier 1/2 failures.
  What farmer does: Wraps a string or thin rope around tree trunk at
                    chest height (~1.3 metres from ground). Marks where
                    it meets itself. Measures that length in centimetres.
                    Enters the circumference in the app.
  App calculates: DBH = circumference / π
  Scientific note: This is identical to the girth tape method used by
                   professional foresters. Fully valid for carbon audits.
  Accuracy:        Depends on farmer care. Grey "Manual Measurement" badge.
  Tutorial:        Short animation showing string wrapping method is shown
                   before the first manual measurement of the session.

TIER DETECTION CODE (Kotlin in ARModule.kt):
  fun checkDepthSupport(): String {
    val availability = ArCoreApk.getInstance()
                        .checkAvailability(context)
    if (availability == Availability.UNSUPPORTED) return "UNSUPPORTED"
    // ARCore available — check depth mode support
    val session = Session(context)
    return if (session.isDepthModeSupported(Config.DepthMode.RAW_DEPTH_ONLY))
           "FULL_DEPTH" else "SLAM_ONLY"
  }

TypeScript mapping:
  detectARTier() returns:
    "FULL_DEPTH"  → store audit.arTier = 1
    "SLAM_ONLY"   → store audit.arTier = 2
    "UNSUPPORTED" → store audit.arTier = 3

8.1 PURPOSE

8.3 LAYER 1 — NISAR L-BAND SAR (Primary from June 2026)
---------------------------------------------------------
What it is: Joint NASA-ISRO dual-band SAR satellite.
Why better: L-band (~24cm wavelength) penetrates the full tree canopy.
            Reaches branches and trunk — where most biomass is stored.
            Sentinel-1 C-band (~6cm) only reaches upper canopy.
            L-band does not saturate in high-biomass forests.
Resolution: 5-10 metres per pixel.
Update rate: Every 6-12 days globally.
Cost:        Free. Data released 1-2 days after acquisition.
Access:      asf_search Python library (Alaska Satellite Facility).
Product:     GCOV (Geocoded Covariance) — designed for biomass estimation.

Bands used:
  HH (horizontal transmit, horizontal receive) — sensitive to trunk and
      large branches.
  HV (horizontal transmit, vertical receive) — sensitive to canopy
      volume.
  HH/HV ratio — captures forest structural complexity.

NISAR data fetch code:
  import asf_search as asf
  from shapely.geometry import shape

  def fetch_nisar_data(geojson_polygon, audit_year):
      polygon = shape(geojson_polygon)
      results = asf.search(
          platform=asf.PLATFORM.NISAR,
          intersectsWith=polygon.wkt,
          processingLevel='GCOV',
          start=f'{audit_year}-01-01',
          end=f'{audit_year}-12-31'
      )
      if not results:
          return None, None, None
      # After download: extract HH/HV bands from HDF5 files
      # Upload to GEE as Earth Engine asset for pixel-level operations
      nisar_img = ee.Image(f'users/terratrust/nisar_{audit_year}')
      hh = nisar_img.select('HH').rename('NISAR_HH')
      hv = nisar_img.select('HV').rename('NISAR_HV')
      ratio = hh.divide(hv).rename('NISAR_HH_HV_RATIO')
      return hh, hv, ratio

8.4 LAYER 2 — SENTINEL-1 C-BAND SAR (Current Primary / Future Secondary)
--------------------------------------------------------------------------
What it is: ESA radar satellite. Operational since 2014.
Why used:   Works through clouds — critical during Indian monsoon season.
            Long time series for multi-year biomass comparison.
            Complements NISAR by providing upper canopy information.
Resolution: 10 metres per pixel.
Update rate: Every 6 days.
Cost:        Free via Google Earth Engine.
MANDATORY:  Apply Lee speckle filter before any calculation. Speckle is
            random noise in SAR that makes individual pixel values
            unreliable. Filtering is not optional.

Bands used: VH (volume scattering — canopy volume, correlated to biomass)
            VV (surface scattering — stems, soil background)
            VH/VV ratio (forest structural complexity index)

GEE code:
  s1 = ee.ImageCollection('COPERNICUS/S1_GRD') \
      .filterBounds(region) \
      .filterDate(f'{year}-01-01', f'{year}-12-31') \
      .filter(ee.Filter.eq('instrumentMode', 'IW')) \
      .filter(ee.Filter.listContains('transmitterReceiverPolarisation', 'VH')) \
      .filter(ee.Filter.listContains('transmitterReceiverPolarisation', 'VV')) \
      .select(['VH', 'VV']) \
      .median()

  # Lee speckle filter (focal median, 3×3 kernel)
  s1_filtered = s1.focal_median(radius=1, kernelType='square', units='pixels')

  s1_vh = s1_filtered.select('VH').rename('S1_VH')
  s1_vv = s1_filtered.select('VV').rename('S1_VV')
  s1_ratio = s1_vh.divide(s1_vv).rename('S1_VH_VV_RATIO')

8.5 LAYER 3 — SENTINEL-2 OPTICAL (Canopy Cover + Vegetation Health)
---------------------------------------------------------------------
What it is: ESA multispectral optical satellite.
Why used:   Provides what radar cannot — visual vegetation information.
            NDVI identifies tree cover vs bare ground.
            EVI performs better than NDVI in high-biomass areas.
            Red-Edge (B7) specifically improves plantation forest accuracy.
Resolution: 10 metres per pixel.
Update rate: Every 5 days.
Cost:        Free via Google Earth Engine.
Limitation: Cannot penetrate clouds. Use annual median to handle this.
Also used:  Sentinel-2 true colour PNG (B4, B3, B2) for BoundaryConfirmScreen
            satellite background image and land parcel thumbnails.

GEE code:
  s2 = ee.ImageCollection('COPERNICUS/S2_SR_HARMONIZED') \
      .filterBounds(region) \
      .filterDate(f'{year}-01-01', f'{year}-12-31') \
      .filter(ee.Filter.lt('CLOUDY_PIXEL_PERCENTAGE', 20)) \
      .median()

  # NDVI: (NIR - Red) / (NIR + Red). Dense trees: 0.6-0.9. Bare: 0.1-0.2
  ndvi = s2.normalizedDifference(['B8', 'B4']).rename('NDVI')

  # EVI: Better than NDVI in high-biomass areas (does not saturate)
  evi = s2.expression(
      '2.5 * (NIR - RED) / (NIR + 6 * RED - 7.5 * BLUE + 1)',
      {'NIR': s2.select('B8'), 'RED': s2.select('B4'),
       'BLUE': s2.select('B2')}
  ).rename('EVI')

  # Red-Edge B7: proven to improve plantation forest accuracy
  red_edge = s2.select('B7').rename('RED_EDGE')

8.6 LAYER 4 — GEDI LIDAR (Canopy Height — PRIMARY HEIGHT SOURCE)
-----------------------------------------------------------------
What it is: NASA's spaceborne LiDAR instrument on the International
            Space Station. Shoots laser pulses at the earth and measures
            the return time to calculate canopy height.
Why used:   Provides actual tree canopy height measured from space.
            More consistent than farmer-measured AR height.
            This is why AR height measurement is only a fallback.
Resolution: 25-metre diameter footprint circles.
Sampling:   Footprints spaced 60m along-track, 600m across-track.
            For small farms, GEDI footprints may not cover every zone.
            Always use AR anchor height as fallback.
Cost:       Free via Google Earth Engine.
Funded:     Active and collecting data until January 2031.
GEE dataset: LARSE/GEDI/GEDI02_A_002_MONTHLY

Metric used: rh98 (98th percentile height = top of canopy)

GEE code:
  gedi = ee.ImageCollection('LARSE/GEDI/GEDI02_A_002_MONTHLY') \
      .filterBounds(region) \
      .select(['rh98']) \
      .mean() \
      .rename('GEDI_HEIGHT')

Height selection logic (Python, per scanned tree):
  def get_tree_height(scan, gedi_image):
      gedi_value = gedi_image.sample(
          region=ee.Geometry.Point([scan['gps']['lng'], scan['gps']['lat']]),
          scale=25
      ).first().getInfo()

      if gedi_value and gedi_value['properties']['GEDI_HEIGHT'] > 0:
          return gedi_value['properties']['GEDI_HEIGHT'], 'GEDI'
      else:
          return scan['height_m'], 'AR_FALLBACK'

8.7 LAYER 5 — SRTM TOPOGRAPHY (Terrain Correction)
----------------------------------------------------
What it is: NASA Shuttle Radar Topography Mission — global 30m elevation
            dataset from a single space shuttle mission in 2000.
Why used:   Two reasons:
            1. Trees on steep slopes have different biomass dynamics
               than trees on flat land. Slope must be accounted for.
            2. Terrain causes systematic radar signal distortion: slopes
               facing toward the satellite appear brighter (looks like
               more biomass). SRTM lets the model correct this.
Resolution: 30 metres per pixel.
Update rate: Static. Does not change. Earth's elevation is stable.
Cost:        Free via Google Earth Engine globally.

GEE code:
  srtm = ee.Image('USGS/SRTMGL1_003')
  elevation = srtm.select('elevation').rename('ELEVATION')
  slope = ee.Terrain.slope(srtm).rename('SLOPE')
  # aspect = ee.Terrain.aspect(srtm).rename('ASPECT')  # Not used in v3.1

================================================================================
SECTION 9: MODULE 5 — BIOMASS CALCULATION (XGBOOST FUSION ENGINE)
================================================================================

9.1 PURPOSE
-----------
Use the AR-scanned ground truth trees to train a machine learning model.
Then use that model to predict biomass for every 10m × 10m pixel in the
verified land boundary. Sum all pixel predictions = total farm biomass.

Why XGBoost instead of a single k-factor:
  k-factor: One number. Assumes VH/Biomass is constant everywhere.
            Fails when forest is heterogeneous (dense + sparse areas).
  XGBoost : 100 decision trees. Learns complex non-linear patterns.
            E.g. "In dense areas where SAR saturates, weight NDVI more.
                  On steep slopes, apply terrain correction.
                  Young plantation vs mature trees behave differently."

9.2 STEP 1 — INDIVIDUAL TREE BIOMASS (CHAVE'S ALLOMETRIC EQUATION)
--------------------------------------------------------------------
For each AR-scanned tree, calculate above-ground biomass.

Formula: AGB = 0.0673 × (ρ × D² × H)^0.976

Where:
  AGB = Above-ground biomass in kilograms (kg) per tree
  ρ   = Wood density in g/cm³ (looked up from species table, Section 7.4)
  D   = Diameter at breast height (DBH) in cm (from AR scan, Tier 1/2/3)
  H   = Tree height in metres (GEDI primary, AR fallback)
  0.0673 and 0.976 = Chave et al. (2014) pantropical constants

Source: Chave et al. (2014), "Improved allometric models to estimate the
aboveground biomass of tropical trees." Global Change Biology, 20(10).

Python implementation:
  def chave_equation(wood_density: float, dbh_cm: float,
                     height_m: float) -> float:
      """Returns above-ground biomass in kg."""
      return 0.0673 * ((wood_density * (dbh_cm ** 2) * height_m) ** 0.976)

  def agb_to_density(agb_kg: float, crown_area_ha: float = 0.01) -> float:
      """Convert AGB kg to tonnes per hectare for XGBoost training.
      crown_area_ha: approx crown area (default 0.01 ha = 100 m2 = 10x10m)"""
      return (agb_kg / 1000) / crown_area_ha

Example calculation:
  Species:      Teak
  DBH:          22.4 cm
  GEDI Height:  14.2 m
  Wood density: 0.60 g/cm³

  AGB = 0.0673 × (0.60 × 22.4² × 14.2)^0.976
      = 0.0673 × (0.60 × 501.76 × 14.2)^0.976
      = 0.0673 × (4275.0)^0.976
      = 0.0673 × 3421.8
      = 230.3 kg per tree (0.2303 tonnes)
      = 23.03 tonnes per hectare at 100 sqm crown area

9.3 STEP 2 — BUILD TRAINING DATASET
-------------------------------------
For each scanned tree:
  a. Get its GPS location.
  b. Extract the 12 satellite feature values at that GPS location.
  c. Calculate its biomass density using Chave equation.
  d. Create one training data point: features → biomass.

GEE code:
  training_points = []
  for scan in tree_scans:
      # Get satellite feature values at this tree's GPS location
      pixel_values = feature_stack.sample(
          region=ee.Geometry.Point([scan['gps']['lng'], scan['gps']['lat']]),
          scale=10
      ).first().getInfo()['properties']

      # Get height: GEDI preferred, AR fallback
      height, height_source = get_tree_height(scan, gedi_image)

      # Calculate biomass
      agb_kg = chave_equation(
          wood_density=SPECIES_DENSITY[scan['species']],
          dbh_cm=scan['dbh_cm'],
          height_m=height
      )
      agb_t_per_ha = agb_to_density(agb_kg)

      training_points.append({
          'features': pixel_values,
          'biomass': agb_t_per_ha,
          'height_source': height_source
      })

Minimum viable training set: 9 trees (3 zones × 3 trees = minimum viable)
Ideal training set:          15+ trees for better model accuracy.

9.4 STEP 3 — TRAIN XGBOOST MODEL ON GEE
-----------------------------------------
GEE has a built-in gradient boosted trees classifier (smileGradientTreeBoost)
that works on GEE FeatureCollections. Use this instead of downloading data
and running scikit-learn locally.

  # Convert training points to GEE FeatureCollection
  training_fc = ee.FeatureCollection([
      ee.Feature(
          ee.Geometry.Point([pt['features']['longitude'],
                             pt['features']['latitude']]),
          {'biomass': pt['biomass'], **pt['features']}
      )
      for pt in training_points
  ])

  # Train XGBoost via GEE
  xgb_model = ee.Classifier.smileGradientTreeBoost(
      numberOfTrees=100,    # 100 decision trees
      shrinkage=0.05,       # Learning rate — smaller = more careful
      samplingRate=0.7,     # Use 70% of training data per tree
      maxNodes=None         # Let tree depth be determined automatically
  ).setOutputMode('REGRESSION').train(
      features=training_fc,
      classProperty='biomass',
      inputProperties=feature_stack.bandNames()
  )

9.5 STEP 4 — PREDICT BIOMASS FOR ENTIRE LAND
----------------------------------------------
Apply the trained model to every 10m pixel inside the land boundary.
Sum all predictions × pixel area = total farm above-ground biomass.

  # Predict biomass density for every pixel (tonnes per hectare per pixel)
  biomass_map = feature_stack.classify(xgb_model)

  # Multiply by pixel area to convert t/ha to tonnes for that pixel
  # Each pixel = 10m × 10m = 100 m2 = 0.01 hectares
  pixel_area_ha = 0.01

  # Sum all pixels within the official land boundary
  total_biomass_result = biomass_map.multiply(pixel_area_ha).reduceRegion(
      reducer=ee.Reducer.sum(),
      geometry=region,                    # Verified land boundary polygon
      scale=10,                           # Match Sentinel resolution: 10m
      maxPixels=1e9
  ).getInfo()

  total_biomass_tonnes = total_biomass_result.get('classification', 0)
  # This is total above-ground biomass (AGB) in tonnes for the entire farm

9.6 STEP 5 — CARBON CREDIT CALCULATION
----------------------------------------
Carbon credits are earned by GROWING more carbon — not just having trees.
A farm with the same biomass as last year earns zero credits.

  def calculate_credits(total_biomass_tonnes: float,
                        land_id: str,
                        audit_year: int) -> dict:

      # Query last year's audit from database
      prev_audit = get_previous_audit(land_id, audit_year - 1)
      prev_biomass = prev_audit['total_biomass_tonnes'] if prev_audit else 0
      # First year: no previous audit means prev_biomass = 0 (baseline year).
      # Credits are still earned in first year if biomass > 0.

      delta_biomass = total_biomass_tonnes - prev_biomass

      if delta_biomass <= 0:
          # No growth or decline — no credits issued
          return {
              'credits_issued': 0,
              'reason': 'No biomass growth vs previous year'
          }

      # IPCC standard carbon fraction: Carbon is 47% of dry biomass
      carbon_tonnes = delta_biomass * 0.47

      # CO2 equivalent: 1 tonne carbon = 44/12 = 3.667 tonnes CO2
      co2_equivalent = carbon_tonnes * 3.667

      # 1 CTT token = 1 tonne CO2 equivalent
      credits_issued = co2_equivalent

      return {
          'prev_biomass':    prev_biomass,
          'current_biomass': total_biomass_tonnes,
          'delta_biomass':   delta_biomass,
          'carbon_tonnes':   carbon_tonnes,
          'co2_equivalent':  co2_equivalent,
          'credits_issued':  credits_issued
      }

SECTION 10: MODULE 6 — BLOCKCHAIN AND TOKEN MINTING

10.1 WHY BLOCKCHAIN
--------------------
Carbon credit buyers need to verify three things:
  1. The credit is real (backed by measured, verified data).
  2. The credit has not been sold to anyone else before.
  3. When they use (retire) a credit, it cannot be used again by anyone.

Polygon blockchain provides all three through its permanent, public,
tamper-proof ledger. Nobody — not even the TerraTrust team — can edit
what is written on it once confirmed.

10.2 WHAT IS CREATED PER AUDIT
--------------------------------
Every completed audit creates TWO tokens on the blockchain in one
transaction (both minted in the same mintAudit() call to save gas fees):

Token Type 1: Carbon Credit Token (CTT) — Fungible ERC-1155
  Token ID:   1 (constant for all carbon credits in the system)
  Quantity:   Number of CO2 tonnes earned (e.g., 12 = 12 tonnes)
  Properties: Interchangeable — 1 CTT from Farm A = 1 CTT from Farm B.
              Can be transferred and sold.
              Can be permanently retired (burned) by buyer.

Token Type 2: Audit Certificate — Non-Fungible ERC-1155
  Token ID:   audit_id converted to integer (unique per audit)
  Quantity:   Always 1 (one certificate per audit)
  Properties: Contains IPFS link to complete evidence package.
              Cannot be transferred — represents a specific audit record.
              Permanently on-chain as proof of audit.

  // SPDX-License-Identifier: MIT
  pragma solidity ^0.8.20;

  import "@openzeppelin/contracts/token/ERC1155/ERC1155.sol";
  import "@openzeppelin/contracts/access/Ownable.sol";

  contract TerraTrustToken is ERC1155, Ownable {

      // Token ID 1 = Carbon Credit (fungible — all credits are identical)
      uint256 public constant CARBON_CREDIT = 1;

      // Store IPFS evidence link for each audit certificate
      // Key: audit_id → Value: "ipfs://CID..."
      mapping(uint256 => string) public auditEvidence;

      // Prevent minting same land + year twice
      // Key: keccak256(landId + auditYear) → already_minted
      mapping(bytes32 => bool) public auditMinted;

      // Track total credits retired (burned) per wallet address
      mapping(address => uint256) public retiredCredits;

      // Public event emitted every time credits are permanently retired
      event CreditRetired(
          address indexed retiredBy,
          uint256 amount,
          string retirementReason,
          uint256 timestamp
      );

      constructor() ERC1155("") Ownable(msg.sender) {}

      // ----------------------------------------------------------------
      // MINT FUNCTION — called ONLY by backend Oracle wallet (onlyOwner)
      // Farmer cannot call this directly. Only the admin wallet can.
      // ----------------------------------------------------------------
      function mintAudit(
          address farmer,         // Farmer's wallet address (0x...)
          uint256 auditId,        // Unique audit ID (UUID converted to int)
          uint256 creditAmount,   // Number of CTT to mint (e.g. 12)
                                  // CRITICAL: Send actual number, NOT *10^18
                                  // Old bug: was multiplied by 10^18 → FIXED
          string memory ipfsHash, // "ipfs://CID..." — evidence package link
          string memory landId,   // Internal land UUID as string
          uint256 auditYear       // e.g. 2026
      ) public onlyOwner {

          // Safety: Prevent minting same land + year twice
          bytes32 auditKey = keccak256(
              abi.encodePacked(landId, auditYear)
          );
          require(
              !auditMinted[auditKey],
              "TerraTrust: Credits already minted for this land this year"
          );
          auditMinted[auditKey] = true;

          // Mint carbon credit tokens to farmer
          _mint(farmer, CARBON_CREDIT, creditAmount, "");

          // Mint unique audit certificate NFT to farmer
          _mint(farmer, auditId, 1, "");

          // Store IPFS evidence link permanently on-chain
          auditEvidence[auditId] = ipfsHash;
      }

      // ----------------------------------------------------------------
      // RETIRE FUNCTION — called by carbon credit BUYER
      // Permanently destroys credits after use for offsetting.
      // Required by all carbon standards (Verra, Gold Standard).
      // ----------------------------------------------------------------
      function retireCredits(
          uint256 amount,
          string memory reason
      ) public {
          require(
              balanceOf(msg.sender, CARBON_CREDIT) >= amount,
              "TerraTrust: Insufficient credits to retire"
          );

          // Permanently destroy — cannot be undone
          _burn(msg.sender, CARBON_CREDIT, amount);

          // Track retirement total for this address
          retiredCredits[msg.sender] += amount;

          // Write permanent public retirement record on blockchain
          emit CreditRetired(msg.sender, amount, reason, block.timestamp);
      }
  }

  import json, os, requests
  from web3 import Web3
  from eth_account import Account

  w3 = Web3(Web3.HTTPProvider(os.getenv('ALCHEMY_POLYGON_AMOY_URL')))
  admin = Account.from_key(os.getenv('ADMIN_WALLET_PRIVATE_KEY'))

  with open('./contracts/artifacts/TerraToken_ABI.json') as f:
      CONTRACT_ABI = json.load(f)

  contract = w3.eth.contract(
      address=w3.to_checksum_address(os.getenv('CONTRACT_ADDRESS')),
      abi=CONTRACT_ABI
  )

  def mint_carbon_credits(farmer_address: str, audit_id_int: int,
                           credit_amount: int, metadata: dict,
                           land_id: str, audit_year: int) -> dict:

      # Step 1: Upload evidence to IPFS
      ipfs_url = upload_to_ipfs(metadata)

      # Step 2: Build transaction
      nonce = w3.eth.get_transaction_count(admin.address)
      gas_price = w3.eth.gas_price

      tx = contract.functions.mintAudit(
          w3.to_checksum_address(farmer_address),
          audit_id_int,
          credit_amount,    # Actual number, NO *10^18
          ipfs_url,
          land_id,
          audit_year
      ).build_transaction({
          'from':     admin.address,
          'nonce':    nonce,
          'gas':      300000,
          'gasPrice': gas_price,
          'chainId':  80002    # Use 137 for mainnet
      })

      # Step 3: Sign and send
      signed_tx = admin.sign_transaction(tx)
      tx_hash = w3.eth.send_raw_transaction(signed_tx.raw_transaction)

      # Step 4: Wait for on-chain confirmation (up to 60 seconds)
      receipt = w3.eth.wait_for_transaction_receipt(tx_hash, timeout=60)

      if receipt['status'] == 0:
          raise Exception(f"Transaction failed: {tx_hash.hex()}")

      return {
          'tx_hash': tx_hash.hex(),
          'ipfs_url': ipfs_url,
          'block_number': receipt['blockNumber']
      }

  RETURNS TRIGGER AS $$
  BEGIN
      -- ST_Area with ::geography cast calculates in square metres
      -- Divide by 10000 to convert to hectares
      NEW.area_hectares := ST_Area(NEW.geom::geography) / 10000;
      RETURN NEW;
  END;

      s2_ndvi_mean     FLOAT,     -- Sentinel-2 NDVI mean over farm
      s2_evi_mean      FLOAT,     -- Sentinel-2 EVI mean over farm
      gedi_height_mean FLOAT,     -- GEDI rh98 mean over farm (metres)
      srtm_elevation_mean FLOAT,  -- SRTM mean elevation (metres)
      nisar_used       BOOLEAN DEFAULT FALSE,
      -- True if NISAR data was included (only from June 2026)
      xgboost_model_version VARCHAR(20),  -- e.g., 'v3.1.0'
      features_count   INT,       -- 9 (pre-NISAR) or 12 (post-NISAR)

      -- Biomass calculation results
      total_biomass_tonnes FLOAT, -- Total AGB of entire farm this year
      prev_year_biomass    FLOAT, -- From previous year's audit
      delta_biomass        FLOAT, -- Growth = current - previous
      carbon_tonnes        FLOAT, -- delta * 0.47 (IPCC carbon fraction)
      co2_equivalent       FLOAT, -- carbon * 3.667 (CO2 conversion)
      credits_issued       FLOAT, -- = co2_equivalent (1 CTT = 1 tonne CO2e)
      trees_scanned_count  INT,

      -- Blockchain records
      ipfs_metadata_cid  VARCHAR(255),
      -- NFT.Storage CID. Full URL = "ipfs://[cid]"
      tx_hash            VARCHAR(66),
      -- Polygon transaction hash (0x + 64 hex = 66 chars)
      token_id           BIGINT,
      -- ERC-1155 audit certificate token ID
      minted_at          TIMESTAMP WITH TIME ZONE,
      created_at         TIMESTAMP WITH TIME ZONE DEFAULT NOW()
  );

  -- Prevent two audits for same land in same year
  CREATE UNIQUE INDEX idx_audits_land_year ON carbon_audits(land_id, audit_year);
  CREATE INDEX idx_audits_user   ON carbon_audits(user_id);
  CREATE INDEX idx_audits_status ON carbon_audits(status);

NOTE: Old SRS v3.0 named this table 'biomass_audits'. v3.1 renames to
      'carbon_audits'. The old table also lacked: nisar_used, user_id,
      features_count, xgboost_model_version, token_id, minted_at fields.

11.7 ROW LEVEL SECURITY (RLS) — MANDATORY
------------------------------------------
Supabase uses PostgreSQL RLS to ensure each farmer can only read and
write their own data. Run these in Supabase SQL Editor:

  ALTER TABLE land_parcels ENABLE ROW LEVEL SECURITY;
  CREATE POLICY "Users own their land" ON land_parcels
      FOR ALL USING (auth.uid()::text = user_id::text);

  ALTER TABLE ar_tree_scans ENABLE ROW LEVEL SECURITY;
  CREATE POLICY "Users own their scans" ON ar_tree_scans
      FOR ALL USING (land_id IN (
          SELECT id FROM land_parcels
          WHERE user_id::text = auth.uid()::text
      ));

  ALTER TABLE carbon_audits ENABLE ROW LEVEL SECURITY;
  CREATE POLICY "Users own their audits" ON carbon_audits
      FOR ALL USING (user_id::text = auth.uid()::text);

  ALTER TABLE sampling_zones ENABLE ROW LEVEL SECURITY;
  CREATE POLICY "Users own their zones" ON sampling_zones
      FOR ALL USING (land_id IN (
          SELECT id FROM land_parcels
          WHERE user_id::text = auth.uid()::text
      ));

-- Geofencing: Is farmer's current GPS inside a specific land boundary?
SELECT id FROM land_parcels
WHERE user_id = '[user_uuid]'
AND ST_Contains(
    geom,
    ST_SetSRID(ST_MakePoint([farmer_lng], [farmer_lat]), 4326)
);

FROM land_parcels WHERE id = '[land_uuid]';

-- Is submitted GeoJSON a valid polygon?

    ST_GeomFromGeoJSON('[geojson_string]')

-- Find all scanned trees within a land boundary:
SELECT * FROM ar_tree_scans
WHERE ST_Within(
    gps_location,
    (SELECT geom FROM land_parcels WHERE id = '[land_id]')
);

-- Get last year's biomass for delta calculation:
SELECT total_biomass_tonnes FROM carbon_audits
WHERE land_id = '[land_uuid]'
ORDER BY audit_year DESC LIMIT 1;

  class ARModule(private val reactContext: ReactApplicationContext)
      : ReactContextBaseJavaModule(reactContext) {

    override fun getName() = "ARModule"
    // This string is how JavaScript accesses the module:
    // NativeModules.ARModule.[method]

    @ReactMethod
    fun checkDepthSupport(promise: Promise) {
        try {
            val availability = ArCoreApk.getInstance()
                .checkAvailability(reactContext)
            if (availability == ArCoreApk.Availability.UNSUPPORTED) {
                promise.resolve("UNSUPPORTED")
                return
            }
            val session = Session(reactContext)
            val supportsDepth = session.isDepthModeSupported(
                Config.DepthMode.RAW_DEPTH_ONLY
            )
            session.close()
            promise.resolve(if (supportsDepth) "FULL_DEPTH" else "SLAM_ONLY")
        } catch (e: Exception) {
            promise.resolve("UNSUPPORTED")
        }
    }

    @ReactMethod
    fun measureCylinder(promise: Promise) {
        // Step 1: Configure ARCore session
        val config = Config(arCoreSession)
        config.depthMode = Config.DepthMode.RAW_DEPTH_ONLY
        // RAW_DEPTH_ONLY: only confident measurements included.
        // More accurate than AUTOMATIC which fills gaps with estimates.
        arCoreSession.configure(config)

        // Step 2: Acquire 16-bit raw depth map
        val depthImage = frame.acquireRawDepthImage16Bits()

        // Step 3: Filter to centre 30% width × 60% height of frame
        //         (reticle area where trunk should be)

        // Step 4: Convert depth pixels to 3D world coordinates
        //         using camera intrinsics (focal length + principal point)

        // Step 5: RANSAC cylinder fitting (100 iterations minimum):
        //   a) Randomly select 2 points to define axis direction
        //   b) Randomly select 1 point to define radius
        //   c) Count inliers (points within epsilon distance of cylinder)
        //   d) Update best cylinder if inlier count is higher
        //   Repeat 100 times.

        // Step 6: confidence = best_inlier_count / total_point_count
        //         If confidence < 0.7 → return error

        // Step 7: Return result as JSON string
        val result = JSONObject().apply {
            put("diameter_cm", bestCylinder.radius * 2 * 100)
            put("confidence", confidence)
            put("tier_used", 1)  // or 2 if SLAM mode
            put("point_count", totalPoints)
        }
        promise.resolve(result.toString())
    }

    @ReactMethod
    fun checkMockLocation(promise: Promise) {
        val isMock = Settings.Secure.getInt(
            reactContext.contentResolver,
            Settings.Secure.ALLOW_MOCK_LOCATION, 0
        ) == 1
        promise.resolve(isMock)
    }
  }

SECTION 15: SECURITY DESIGN
================================================================================

THREAT: Farmer scans trees outside their registered land
PROTECTION:
  useGeofence hook checks real-time GPS against PostGIS land boundary
  using ST_Contains() query on every GPS update during audit.
  Scanning button is disabled if GPS is outside the boundary.
  Message: "You appear to be outside your registered land. Please
  return to your land to continue scanning."
  Zone arrival check: scan button locked until farmer is within 10
  metres of zone centre GPS. Prevents remote or desk-based scanning.

THREAT: Farmer spoofs GPS location
PROTECTION:
  Kotlin code checks mock location setting on Android:
    Settings.Secure.getInt(contentResolver,
        Settings.Secure.ALLOW_MOCK_LOCATION, 0) == 1
  If mock location is enabled:
    Block audit start screen with message:
    "Please disable Mock Location in Developer Settings to use this app."
    Audit cannot start until mock location is off.
  Developer Options check:
    react-native-device-info isEmulator() blocks on emulators.

THREAT: Evidence photo tampered with after upload
PROTECTION:
  SHA-256 hash of each evidence photo computed on-device BEFORE upload.
  Hash sent to backend with photo. Hash stored on IPFS metadata.
  Hash stored in ar_tree_scans.evidence_photo_hash.
  Any post-upload modification changes the hash — mismatch = fraud.
  Code (src/common/utils/hash.ts):
    import { createHash } from 'react-native-quick-crypto';
    export const hashPhoto = (base64: string): string =>
        createHash('sha256').update(base64).digest('hex');

THREAT: Farmer's private key exposure
PROTECTION:
  Private key stored ONLY in react-native-keychain using Android Keystore.
  Hardware-backed encrypted storage — key never leaves the device.
  Key never stored in Redux state.
  Key never stored in MMKV or AsyncStorage.
  Key never logged to console.
  Key never sent to any server under any circumstances.
  Only the public wallet address is sent to backend.

THREAT: JWT token interception
PROTECTION:
  Supabase JWTs expire after 1 hour and auto-refresh.
  Stored in MMKV (encrypted) not AsyncStorage (plain text).
  All API requests over HTTPS only — no HTTP in production.

THREAT: App running on rooted device
PROTECTION:
  react-native-device-info isRooted() check on startup.
  If rooted: Show warning banner. App continues — not blocked.
  Some legitimate users use custom ROMs for valid reasons.
  WARNING only — scanning proceeds normally.

THREAT: Backend admin private key exposure
PROTECTION:
  ADMIN_WALLET_PRIVATE_KEY stored in environment variable only.
  Never in code files. Never in git.
  In production: use Render.com Environment Variables panel.
  In development: .env file in .gitignore from the first commit.

THREAT: Smart contract double-minting
PROTECTION:
  bytes32 auditKey = keccak256(abi.encodePacked(landId, auditYear))
  mapping(bytes32 => bool) public auditMinted
  Second mintAudit call with same landId + year reverts with:
  "TerraTrust: Credits already minted for this land this year"

THREAT: SQL injection
PROTECTION:
  All database queries use parameterised Supabase client calls.
  SQLAlchemy ORM with bound parameters for raw queries.
  No string concatenation in SQL ever.

THREAT: GEE service account key exposure
PROTECTION:
  gee-service-account.json in .gitignore from first commit.
  In production: write JSON content to temp file from env variable.
    import tempfile, json
    gee_key = json.loads(os.getenv('GEE_SERVICE_ACCOUNT_JSON'))
    with tempfile.NamedTemporaryFile(mode='w', suffix='.json',
                                     delete=False) as f:
        json.dump(gee_key, f)
        key_path = f.name
    ee.Initialize(ee.ServiceAccountCredentials(email, key_file=key_path))

                           |          | Stored in MMKV. Works offline after first
                           |          | load. Shown in BoundaryConfirmScreen and
                           |          | LandListScreen thumbnails.
Tree data saving           | YES      | Saved to Redux + MMKV immediately.
                           |          | No internet dependency for local save.
Audit submission           | QUEUED   | react-native-background-fetch detects
                           |          | connectivity restore and auto-uploads.
Document OCR               | NO       | Requires backend PaddleOCR server.
                           |          | Inform farmer to retry with internet.
Boundary API fetch         | NO       | Requires WMS API call to government server.
Blockchain balance         | CACHED   | Last known balance shown from MMKV cache.
                           |          | Refreshed when online.

BACKGROUND SYNC IMPLEMENTATION:
When farmer submits audit data with no internet, store payload in MMKV
under key 'pending_upload'. react-native-background-fetch triggers when
connectivity is restored and calls POST /api/v1/audit/submit-samples.

Multi-day audit handling:
  Farms >10 acres: Zones split into 2 sessions on separate days.
  All Session 1 scan data persisted in MMKV.
  On Session 2 start: app resumes at the first uncompleted zone.
  Session 1 data intact. No restarting.

SECTION 23: ERROR HANDLING STRATEGY
================================================================================

23.1 OCR FAILURE (PaddleOCR)
------------------------------
Trigger:   PaddleOCR cannot extract survey number or owner name from image.
Response:  "The document photo is unclear. Please:
           - Use bright, even lighting (avoid shadows on document)
           - Hold phone steady and parallel to the document
           - Ensure all text is visible and not cut off
           Try again"
Retry:     Allow unlimited retries. Never block farmer at this step.
Note:      Old SRS Section 16.1 said "Tesseract cannot read" — CORRECTED
           to PaddleOCR in v3.1.

23.2 LGD API FAILURE
---------------------
Trigger:   Government API at 115.124.105.220 is down, slow, or returns error.
Response:  Retry 3 times with 2-second delays using exponential backoff.
           If all 3 fail: silently fallback to Layer 2 (scraping).
           Do NOT show farmer a technical error. Show Layer 2 transparently.
Log:       Record failure with timestamp, state, district for monitoring.

23.3 WMS GETFEATUREINFO FAILURE
---------------------------------
Trigger:   BhuNaksha portal returns empty response, timeout, or HTTP error.
Response:  Silently fallback to Layer 3 manual upload.
           Log: state + district + taluka + survey_number + HTTP status code.
Monitor:   If failure rate for a state exceeds 50% → remove from Tier 2 list.

23.4 GEE PROCESSING FAILURE
-----------------------------
Trigger:   Earth Engine returns timeout, rate limit, or computation error.
Response:  Celery retries with exponential backoff:
             Attempt 1: Immediate
             Attempt 2: After 5 minutes
             Attempt 3: After 30 minutes
           After 3 failures: Update audit status to 'FAILED'.
           Farmer sees: "Satellite processing delayed. Please check back
           in 1 hour. Your scan data is saved."
           Admin alert: Log audit_id for manual re-trigger.

23.5 BLOCKCHAIN TRANSACTION FAILURE
-------------------------------------
Trigger:   Transaction rejected due to gas too low or network congestion.
Response:  Increase gas price by 20% and retry once automatically.
           If second attempt fails: Set status = 'FAILED', queue for retry.
           Farmer sees: "Token minting delayed. Your credits are calculated
           and will be issued within 24 hours."
           Credit amount is NEVER lost — always retryable from database.

23.6 AR MEASUREMENT FAILURE
-----------------------------
Trigger:   RANSAC confidence < 0.7, or point cloud < 50 points.
Response:  "Not enough depth data. Please:
           - Move to 1-2 metres from the tree trunk
           - Keep the trunk centred in the crosshair
           - Hold your phone steady and try again"
Auto-fallback: After 3 consecutive AR failures in one session,
               automatically switch to Tier 3 (manual string entry).

23.7 GPS ACCURACY TOO LOW
---------------------------
Trigger:   GPS accuracy reading > 20 metres (poor signal).
Response:  "GPS signal is weak. Please:
           - Move away from buildings or dense tree canopy
           - Wait 30 seconds for GPS to stabilise
           - Check Location mode is set to 'High Accuracy' in settings"
Block:     Do not allow tree scan to be saved if GPS accuracy > 30 metres.

23.8 OWNER NAME MISMATCH
--------------------------
Trigger:   OCR-extracted owner name fuzzy-match score < 80% vs KYC name.
Response:  "The name on this document does not match your registered name.
           Please use the land document where you are listed as the owner,
           or contact support if you need to update your registered name."
Do not:   Reveal what the KYC name is in the error (privacy).

SECTION 28: GLOSSARY
================================================================================

AGB         Above-Ground Biomass. Dry weight of all plant material above
            the soil surface, in kg or tonnes.

Alchemy     Free RPC provider for connecting Python backend to Polygon
            blockchain without running your own node.

ARCore      Google's augmented reality framework for Android. Provides
            depth maps, SLAM tracking, and 3D point cloud data.

AGB density AGB in tonnes per hectare. Used as the training target
            variable for XGBoost.

ASF         Alaska Satellite Facility — the NASA distribution centre for
            NISAR L-band satellite data (asf.alaska.edu).

BhuNaksha   National Center for Land Records Management (NIC) web portal
            providing cadastral maps for most Indian states.

Celery      Python distributed task queue. Runs satellite processing and
            blockchain minting as background jobs.

Chave Equation Pantropical allometric equation (Chave et al. 2014) for
               estimating AGB from diameter, height, and wood density:
               AGB = 0.0673 × (ρ × D² × H)^0.976

CTT         Carbon Ton Token. The ERC-1155 fungible token issued on
            Polygon. 1 CTT = 1 tonne CO2 equivalent absorbed.

DBH         Diameter at Breast Height. Tree trunk diameter measured
            at 1.3 metres from ground. Standard forestry measurement.

ERC-1155    Ethereum Multi-Token Standard. Allows both fungible tokens
            (credits) and NFTs (certificates) in one smart contract.

EVI         Enhanced Vegetation Index. Optical vegetation index that
            performs better than NDVI in high-biomass forests.

FastAPI     Modern async Python web framework. Auto-generates Swagger
            documentation at /docs.

GEE         Google Earth Engine. Free cloud platform for planetary-scale
            satellite data processing.

GEDI        Global Ecosystem Dynamics Investigation. NASA LiDAR on the
            International Space Station. Provides rh98 canopy height.

gis_code    NIC internal plot identifier used to query BhuNaksha WMS.
            Constructed from LGD location codes + survey number.

IPFS        InterPlanetary File System. Decentralised content-addressed
            storage. Files stored by hash — content cannot change.

IPCC        Intergovernmental Panel on Climate Change. Standard carbon
            fraction value used: 0.47 (47% of dry biomass is carbon).

LGD         Local Government Directory. Official Indian government database
            of location codes maintained by NIC.

MMKV        Key-value storage library. 30x faster than AsyncStorage.
            Synchronous. Encrypted. Used for AR scan data persistence.

NDVI        Normalized Difference Vegetation Index = (NIR-Red)/(NIR+Red).
            Range: -1 to 1. Dense trees: 0.6-0.9.

NFT.Storage Free permanent IPFS storage service. Files backed by Filecoin.

NISAR       NASA-ISRO Synthetic Aperture Radar. Joint US-India satellite.
            Launched July 30, 2025. L-band SAR for biomass estimation.
            Fully calibrated data: June 2026.

NIC         National Informatics Centre — maintains India's BhuNaksha
            cadastral mapping portal and LGD directory.

OpenZeppelin Library of audited Solidity smart contract templates.
             Used for ERC-1155 and Ownable base contracts.

Polygon     EVM-compatible Layer-2 blockchain. Low gas fees.
            Polygon PoS is used (not zkEVM).
            Amoy testnet (chainId 80002), Mainnet (chainId 137).

PostGIS     PostgreSQL extension that adds geographic objects.
            Enables spatial queries (contains, area, distance).

PaddleOCR   Deep learning OCR engine by Baidu. PP-OCRv5 with Devanagari
            model achieves F1=0.938 on Indic scripts vs Tesseract's 0.797.

RANSAC      Random Sample Consensus. Algorithm for fitting a geometric
            model (cylinder) to noisy 3D point cloud data.

rh98        GEDI metric: 98th percentile height of return waveform.
            Represents the top of tree canopy in metres.

RLS         Row Level Security. PostgreSQL feature ensuring each user
            can only access their own data in Supabase.

SAR         Synthetic Aperture Radar. Radar that works through clouds.
            Sentinel-1 (C-band) and NISAR (L-band) are both SAR.

SLAM        Simultaneous Localisation and Mapping. Algorithm that builds
            a 3D map from camera motion. Used for Tier 2 AR measurement.

SRTM        Shuttle Radar Topography Mission. Global 30m elevation data.
            Used for terrain correction in biomass modelling.

Supabase    Open-source Firebase alternative. Provides PostgreSQL +
            PostGIS + Phone OTP Auth + File Storage. Free tier.

ToF         Time-of-Flight sensor. Hardware depth sensor that measures
            distance using infrared light pulses. Enables Tier 1 AR.

WMS         Web Map Service. OGC standard API for geospatial data.
            BhuNaksha uses WMS GetFeatureInfo to return plot GeoJSON.

XGBoost     Extreme Gradient Boosting. Ensemble ML algorithm. Uses 100
            decision trees to learn non-linear satellite→biomass mapping.

7/12 Extract  Government land record document in Maharashtra showing
              survey number, owner name, crop type. Used for OCR-based
              land ownership verification. Also called "Satbara Utara."
